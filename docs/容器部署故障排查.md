# 🐳 容器部署常见故障排查文档

> **项目**: 算法过程可视化系统  
> **部署方式**: Docker + docker-compose 三容器编排  
> **最后更新**: 2026-06-24

---

## 一、部署架构速览

```
docker compose up -d
  ├── mysql       (MySQL 8.0)    — 端口 3306, 健康检查: mysqladmin ping
  ├── backend     (Node.js)      — 端口 3001, 健康检查: GET /api/health
  └── frontend    (Nginx)        — 端口 80,   依赖 backend healthy
```

启动顺序：`mysql → (healthy) → backend → (healthy) → frontend`

## 二、启动阶段故障

### 故障 2.1：`docker compose up` 直接报错退出

**症状**：
```
ERROR: Missing mandatory value for "DB_PASSWORD" option in service "mysql"
```

**原因**：`docker-compose.yml` 使用 `${DB_PASSWORD:?请设置 DB_PASSWORD 环境变量}` 强制校验，未设置环境变量时拒绝启动。

**解决**：

```bash
# 方式一：命令行传入
DB_PASSWORD=your_password JWT_SECRET=your_secret docker compose up -d

# 方式二：创建 .env 文件（推荐）
cat > .env << 'EOF'
DB_PASSWORD=your_secure_password
JWT_SECRET=your_jwt_secret_here
DB_NAME=algorithm_viz
EOF

docker compose up -d
```

> ⚠️ `.env` 文件包含敏感信息，**不要提交到 Git**（已在 `.gitignore` 中）。

### 故障 2.2：MySQL 容器反复重启

**症状**：
```
algoviz-mysql    restarts: 3 times in 30 seconds
docker compose ps 显示 mysql 状态为 Restarting
```

**排查步骤**：

```bash
# 1. 查看 MySQL 日志
docker compose logs mysql --tail=50

# 2. 常见错误及对应解决
```

| 日志关键字 | 原因 | 解决 |
|-----------|------|------|
| `Access denied for user 'root'` | 密码不匹配 | 确认 `.env` 中 `DB_PASSWORD` 与首次部署时一致。如果更换密码，需先清空数据卷 `docker compose down -v` |
| `Table 'algorithm_viz.xxx' doesn't exist` | init.sql 未执行或执行失败 | 检查 `./server/init.sql` 是否存在且语法正确 |
| `Port 3306 already in use` | 宿主机 3306 端口被占用 | `netstat -ano | findstr 3306` 查占用进程，停止或改端口 |
| `chown: changing ownership` 后卡住 | Windows/macOS 挂载卷权限 | 临时绕过：注释掉 `volumes` 中宿主机挂载，仅用 named volume |

### 故障 2.3：Backend 容器启动后 healthcheck 失败

**症状**：
```
algoviz-backend   health: starting → unhealthy
前端容器等待 backend healthy 超时
```

**排查步骤**：

```bash
# 1. 查看后端日志
docker compose logs backend --tail=100

# 2. 手动测试健康检查
docker exec algoviz-backend node -e "require('http').get('http://localhost:3001/api/health', (r) => {console.log(r.statusCode)})"

# 3. 常见错误
```

| 日志关键字 | 原因 | 解决 |
|-----------|------|------|
| `ECONNREFUSED mysql:3306` | MySQL 未就绪或网络不通 | 确认 MySQL healthcheck 通过后再等 10 秒，后端依赖 `condition: service_healthy` |
| `getaddrinfo ENOTFOUND mysql` | 容器网络问题，DNS 解析失败 | `docker network ls` 确认 compose 网络存在，`docker compose down && docker compose up -d` 重建 |
| `ER_ACCESS_DENIED_ERROR` | 数据库用户名/密码错误 | 检查 `.env` 中 `DB_PASSWORD` 与 MySQL 初始化密码一致 |
| `ER_BAD_DB_ERROR` | 数据库 `algorithm_viz` 不存在 | 确认 `init.sql` 正确挂载到了 `/docker-entrypoint-initdb.d/` |
| `SyntaxError: Cannot use import statement` | Node 版本不支持 ESM | 确认 `backend/Dockerfile` 使用 `node:18-alpine` 或更高版本 |

---

### 故障 2.4：Frontend Nginx 返回 502 Bad Gateway

**症状**：浏览器访问 `http://服务器IP` 显示 `502 Bad Gateway`

**排查**：

```bash
# 查看 Nginx 日志
docker compose logs frontend --tail=50

# 测试后端是否可达
docker exec algoviz-frontend wget -qO- http://backend:3001/api/health
```

常见原因：
- **Backend 未启动或 healthcheck 未通过** → 等后端 healthy 后再试
- **Nginx 配置中 `proxy_pass http://backend:3001` 拼写错误** → 确认 `frontend/nginx.conf` 中 `backend` 服务名与 compose 一致

## 三、运行时故障

### 故障 3.1：API 请求返回 500 错误

**排查步骤**：

```bash
# 查看后端实时日志
docker compose logs -f backend

# 进入容器检查
docker exec -it algoviz-backend sh
cat /app/logs/server-$(date +%Y-%m-%d).log   # 查看当天日志文件
```

| 错误信息 | 原因 | 解决 |
|---------|------|------|
| `ER_NO_SUCH_TABLE: Table 'algorithm_viz.users' doesn't exist` | init.sql 未成功执行 | 可能首次启动时数据目录非空，手动执行：`docker exec -i algoviz-mysql mysql -uroot -p$DB_PASSWORD < server/init.sql` |
| `JSON Web Token is expired` | 用户 token 过期 | 前端自动跳转登录页，用户重新登录即可 |
| `connect ECONNREFUSED` | 数据库连接池耗尽 | `db.js` 中 `connectionLimit: 10`，检查是否有慢查询占用连接 |

### 故障 3.2：容器运行正常但数据在重启后丢失

**症状**：`docker compose down && docker compose up -d` 后用户数据、学习记录全部消失。

**原因**：使用了 `docker compose down -v`（`-v` 会删除 named volume）。

**预防与恢复**：

```bash
# 确认 volume 是否存在
docker volume ls | grep algoviz

# 定期备份数据库到宿主机 ./backups 目录
docker exec algoviz-mysql sh -c \
  'mysqldump -uroot -p$MYSQL_ROOT_PASSWORD algorithm_viz' \
  > ./backups/algoviz_$(date +%Y%m%d_%H%M%S).sql

# 恢复备份
docker exec -i algoviz-mysql mysql -uroot -p$MYSQL_ROOT_PASSWORD algorithm_viz \
  < ./backups/algoviz_20260624_120000.sql
```

> 📌 上一轮优化已将 volume 命名为 `algoviz_mysql_data` 并新增了 `./backups` 挂载目录，备份文件直接落在宿主机 `d:/ks/backups/`。

### 故障 3.3：前端页面白屏或 JS 加载失败

**排查**：

```bash
# 检查 Nginx 是否正确托管了静态文件
docker exec algoviz-frontend ls -la /usr/share/nginx/html/

# 确认 index.html + assets 目录存在
# 如果为空，说明构建阶段失败
```

常见原因：
- **Vite 构建失败** → 查看 `docker compose logs frontend` 中的构建阶段输出
- **浏览器缓存** → 强制刷新 `Ctrl+Shift+R`，或检查 Nginx 静态资源缓存配置（30 天过期）

## 四、性能问题

### 故障 4.1：API 响应越来越慢

**排查**：

```sql
-- 进入 MySQL 检查慢查询
docker exec -it algoviz-mysql mysql -uroot -p$DB_PASSWORD algorithm_viz

-- 检查 learning_records 表行数
SELECT COUNT(*) FROM learning_records;

-- 确认索引存在
SHOW INDEX FROM learning_records;
SHOW INDEX FROM login_records;
SHOW INDEX FROM users;
```

每一张表的索引清单见下表（当前版本已全部建立）：

| 表 | user_id 索引 | 时间戳索引 | 复合索引 |
|----|-------------|-----------|----------|
| `users` | PK(`id`) | `idx_user_created(created_at)` | — |
| `learning_records` | 左前缀命中 | 左前缀命中 | `(user_id, algorithm_id)` / `(user_id, created_at)` / `(user_id, algorithm_name)` |
| `login_records` | 左前缀命中 | `idx_login_at(login_at)` | `(user_id, login_at)` |
| `user_stats` | PK(`user_id`) | — | — |

## 五、常用排查命令速查表

```bash
# ========= 状态查看 =========
docker compose ps                          # 所有容器状态
docker compose ps -a                       # 含已停止容器
docker stats                               # CPU/内存实时使用率

# ========= 日志查看 =========
docker compose logs --tail=100             # 所有容器最近 100 行
docker compose logs -f backend             # 跟踪后端日志
docker compose logs --tail=50 mysql        # MySQL 日志
docker compose logs frontend 2>&1 | grep -i error  # 搜索错误

# ========= 进入容器 =========
docker exec -it algoviz-mysql bash         # 进入 MySQL 容器
docker exec -it algoviz-backend sh         # 进入后端容器 (Alpine 用 sh)
docker exec -it algoviz-frontend sh        # 进入 Nginx 容器

# ========= 网络排查 =========
docker network ls                          # 查看 compose 网络
docker inspect algoviz-backend | grep -A10 Networks  # 查看容器 IP
docker exec algoviz-backend wget -qO- http://mysql:3306  # 测试容器间连通

# ========= 数据库操作 =========
docker exec -it algoviz-mysql mysql -uroot -p$DB_PASSWORD algorithm_viz  # 进入 MySQL Shell
docker exec algoviz-mysql mysqldump -uroot -p$DB_PASSWORD algorithm_viz > backup.sql  # 备份
docker exec -i algoviz-mysql mysql -uroot -p$DB_PASSWORD algorithm_viz < backup.sql  # 恢复
docker volume inspect algoviz_mysql_data   # 查看数据卷位置

# ========= 重建与清理 =========
docker compose down                        # 停止所有容器（保留 volume）
docker compose down -v                     # ⚠️ 停止并删除 volume（数据全丢！）
docker compose up -d --build               # 重新构建并启动
docker compose up -d --build --no-cache    # 完全无缓存重建
docker system prune -a                     # 清理未使用的镜像/容器/网络

# ========= 单容器重启 =========
docker compose restart backend             # 重启后端（不重建镜像）
docker compose up -d --force-recreate backend  # 强制重建后端容器
```

## 六、部署检查清单

每次部署前逐项确认：

- [ ] `.env` 文件存在且包含 `DB_PASSWORD` 和 `JWT_SECRET`（不能使用默认值）
- [ ] 宿主机 80 / 3306 / 3001 端口未被占用
- [ ] `server/init.sql` 文件存在且语法正确
- [ ] Docker 版本 ≥ 20.10，docker-compose 版本 ≥ 2.0
- [ ] 服务器磁盘空间 ≥ 2GB（MySQL 数据和 Docker 镜像）
- [ ] 防火墙已开放 80 端口（如使用云服务器，检查安全组规则）
- [ ] 首次部署后在 `./backups/` 目录执行一次 `mysqldump` 做基线备份

## 七、紧急恢复流程

如果部署完全崩溃需要从零恢复：

```bash
# 1. 定位备份文件
ls -la ./backups/

# 2. 彻底清理（⚠️ 仅在确认有备份时执行）
docker compose down -v

# 3. 重新部署
DB_PASSWORD=xxx JWT_SECRET=yyy docker compose up -d

# 4. 等待所有容器 healthy
docker compose ps   # 全部显示 healthy 或 Up

# 5. 恢复数据
docker exec -i algoviz-mysql mysql -uroot -p$DB_PASSWORD algorithm_viz \
  < ./backups/algoviz_20260624_120000.sql

# 6. 验证
curl http://localhost/api/health
```
